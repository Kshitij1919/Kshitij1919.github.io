Then came a different kind of challenge: learning to think before writing code. I started drawing UML diagrams, thinking about data flow, approaching mechanics as a tools programmer would. I'm still finding my way through that phase, but I've made peace with the fact that it's where real growth lives. I play games obsessively, I love the moment a system finally feels right, and I still get unreasonably excited when the engine already has what I thought I'd have to build myself.

Action / PC Koolgen Interactives · 2023–2024

Tridasha

Action-adventure · Third-person · Unreal Engine 5 · Showcased at IGDC 2023

Tridasha needed a character that felt grounded during open-world traversal and threatening in combat — without either state bleeding into the other visually. I owned the animation architecture and stealth system end-to-end, mostly solo.

Animation Layers: The Animation Blueprint used a state-driven Linked Anim Layer approach. A Boolean flag (bIsInCombat) swapped which layer was active — base Locomotion for traversal, Combat Locomotion for armed states — so each could be tuned independently without breaking the other. Transitions stayed at a stable 60fps because upper-body and lower-body poses never competed for the same blend space.

Stealth & Assassination: Detection layered line-of-sight traces with noise events (footsteps, weapon draws), so enemies reacted to sound before sight. The assassination prompt used a dot product check against the enemy's forward unit vector — a result between 0.5 and 1.0 confirmed the player was in the rear arc. But player intent came first: if the combat weapon was already drawn, the system respected that state and forced a normal attack, even from behind. No unfair auto-executes.

Sports / Mobile Nextwhat Games · 2025 ✔ Live on Play Store

CBG Cricket Battleground

Multiplayer Cricket · Android · Unreal Engine 5 · Backend-Integrated

⚠ Note: The Play Store build reflects the current public version. Auth and RMG systems I built were removed following regulatory changes in India — core gameplay remains intact.

My ownership on CBG was the backend integration layer — building the C++ systems that connected UE5 to a live REST backend handling auth, player data, and real-money currency flows.

Auth & Session Persistence: After OTP login, the server response returned an AccessToken and CustomerID which I persisted to a USaveGame slot. On relaunch, the game reads the slot and fires an auto-login validation API — only restoring the session if the server confirms the token is still valid.

Layered API Architecture: HTTP calls never touched gameplay classes directly. A dedicated helper class handled request construction and two-step response validation (HTTP status + JSON body code). A manager layer owned raw HTTP callbacks, while a controller layer exposed Blueprint-assignable delegates so UI could react to async responses without knowing HTTP existed.

On-Demand Data Fetch: Player stats and match history were intentionally not loaded at game start. Each UI section fired its own API only when opened — keeping startup clean and avoiding redundant network calls for data the player may never view.

Plugin / Tool UE5 · C++ · MIT License v0.1-beta

Modular Save System

UE5 Plugin · GameInstance Subsystem · Blueprint & C++ Compatible

Every UE project eventually needs a save system, and most reach for the same boilerplate: load the slot, cast to the save game class, modify properties, save it back. Modular Save System replaces that ritual with single function calls per actor — and solves the harder problem of keeping save data valid when implementations change.

Three-Layer Architecture: ISaveableInterface defines the contract without the subsystem ever casting to a concrete actor type. USaveSubsystem (a GameInstance Subsystem) is the sole entry point for both C++ and Blueprint users. USaveGameData is plugin-private — a dumb container holding a TMap<FGuid, FActorSaveData> that knows nothing about the game world.

Per-Actor Versioning: Instead of a global version that invalidates the entire save file, each actor carries its own version number inside FActorSaveData — a struct wrapper that also satisfies Unreal's no-nested-TMap UPROPERTY restriction. A format change on one actor doesn't break save data for any other.

FGuid as the Save Key: Actors are identified by ActorGuid, not by name or class. Names aren't unique. ActorInstanceGuid is transient and evaluates to zero at runtime. ActorGuid is stable across sessions and compile builds — the only reliable key for persistent actor identification.
